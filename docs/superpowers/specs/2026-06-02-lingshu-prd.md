## 1. 产品概述

### 1.1 产品愿景

**让每个角色都有自己的声音。**

聆书是一款多角色分音色有声书阅读器。它利用 LLM 自动识别小说中的角色，通过 Qwen3-TTS VoiceDesign 为每个角色生成独一无二的音色，让用户在听书时能够像听广播剧一样区分不同角色——无需人工标注，无需配音演员，全自动完成。

### 1.2 一句话描述

打开 EPUB，等待片刻，听到一本广播剧。

### 1.3 产品定位

| 维度 | 定位 |
|------|------|
| 品类 | 多角色 AI 有声书阅读器 |
| 差异化 | LLM 自动角色识别 + 文字描述生成独特音色，无需录音、无需手动配置 |
| 技术栈 | Flutter 跨平台，OpenAI 兼容 API 驱动 LLM + TTS |
| 目标平台 | Linux 桌面 (MVP 首发) → HarmonyOS 5.0+ → Android / iOS |
| 授权 | 开源 (GPL v3) |

### 1.4 核心价值主张

```
传统听书:  "打开 → 一个声音从头读到尾"
聆书:     "打开 → 等30秒 → 听到林风用热血的少年音、苏婉用温柔的少女音、
           沧明师父用沉稳的老者音... 每个角色都有自己的声音"
```

| 对比维度 | 传统 TTS 听书 | 真人配音有声书 | 聆书 |
|----------|:-----------:|:-----------:|:----:|
| 多角色音色 | ❌ 单人声 | ✅ | ✅ |
| 自动化 | ✅ | ❌ 需配音演员 | ✅ 全自动 |
| 成本 | 免费 | 数千-数万元/本 | LLM ¥0.12/本起 + TTS ¥0/本起 (详见 §9.4) |
| 覆盖范围 | 任意文本 | 仅热门书籍 | 任意 EPUB/TXT |
| 离线可用 | ✅ | ✅ | ⚠️ 需 TTS 服务 |

## 2. 目标用户

### 2.1 用户画像

#### 画像 A：网络小说重度读者

| 属性 | 描述 |
|------|------|
| 年龄 | 18-35 岁 |
| 行为 | 每天阅读 2-6 小时，追更 5+ 本书 |
| 痛点 | 长时间看屏幕眼睛疲劳；通勤/运动时想"看"书但不方便 |
| 场景 | 地铁/公交通勤，睡前，家务时 |
| 期望 | 不同角色的声音要有辨识度，不能都是一个调 |

#### 画像 B：轻小说/日漫爱好者

| 属性 | 描述 |
|------|------|
| 年龄 | 16-30 岁 |
| 行为 | 阅读日本轻小说、web 小说 |
| 痛点 | 轻小说以对话为主，单一声线完全无法区分角色 |
| 场景 | 宅家阅读，想要沉浸式体验 |
| 期望 | 一人称和語尾的口调差异能被准确识别和还原 |

#### 画像 C：视障/阅读障碍用户

| 属性 | 描述 |
|------|------|
| 年龄 | 全年龄段 |
| 行为 | 依赖 TTS 获取文字信息 |
| 痛点 | 现有 TTS 全部单一音色，难以跟踪多角色对话 |
| 场景 | 日常信息获取、学习、娱乐 |
| 期望 | 多角色音色本质上是一种"听觉信息分层"，极大提升理解效率 |

### 2.2 典型使用场景

```
场景 1：日常听书
  → 用户在地铁上打开聆书 → 选择昨天在读的《刀剑神域》
  → 缓存命中，秒进 → 点击播放，听到桐人（少年音）和亚丝娜（少女音）的对话
  → 到站，自动保存进度

场景 2：新书首次打开
  → 用户下载了一本新的 TXT 网文 → 打开聆书 → 导入
  → 弹出提示"正在分析角色..." → 30 秒后分析完成
  → 自动开始朗读，每个角色各有音色
  → 用户发现反派的音色不够阴冷，进入角色设置手动调整

场景 3：离线听书
  → 用户明天要坐长途飞机 → 打开聆书 → 对常看的 3 本书选择"预合成"
  → 几小时后合成完成 → 飞行中全程离线收听

场景 4：微调角色
  → 用户听完第一章 → 觉得女主声音太成熟
  → 进入角色管理页 → 修改女主的音色描述为"16岁少女，声音更青春活泼"
  → 重新合成该角色的对话 → 继续听
```

## 3. 功能需求

### 3.1 功能分级说明

| 优先级 | 含义 |
|:------:|------|
| **P0** | MVP 必须实现，缺少则产品不可用 |
| **P1** | 重要功能，MVP 后续版本尽快补齐 |
| **P2** | 锦上添花，根据反馈决策 |

### 3.2 核心功能矩阵

#### P0 — MVP 必做

| 编号 | 功能 | 描述 |
|:----:|------|------|
| F1 | **EPUB/TXT 导入解析** | 支持打开 EPUB 和 TXT 文件；EPUB 保留章节结构；TXT 自动按章节标记分段 |
| F2 | **LLM 角色分析** | 打开书籍后自动调用 LLM API 分析文本：发现角色、推断属性（性别/年龄/性格）、生成中文音色描述、标注对话归属；支持中日文小说 |
| F3 | **角色音色配置存储** | 分析结果（角色列表 + 音色描述 + 对话脚本）本地持久化存储，二次打开同一本书直接读取缓存 |
| F4 | **多角色朗读** | 逐句朗读，根据脚本自动切换对应角色的音色；旁白使用独立音色；朗读时文本逐句高亮 |
| F5 | **播放控制** | 播放/暂停、上一句/下一句、快进/快退、倍速（0.5x-3x）、章节跳转 |
| F6 | **阅读进度保存** | 自动保存当前朗读位置（书籍+章节+句子索引）；下次打开恢复到上次位置 |
| F7 | **API 配置** | 用户配置 LLM API（URL + Key + Model）和 TTS API（URL + Key + Model）；支持预设（阿里云百炼、本地 Docker） |

#### P1 — 重要功能

| 编号 | 功能 | 描述 |
|:----:|------|------|
| F8 | **角色管理** | 查看本书所有角色列表；预览每个角色的示例语音；手动修改角色的音色描述（VoiceDesign prompt）；添加/删除角色 |
| F9 | **文本阅读模式** | 除朗读外的静默阅读模式；传统翻页/滚动阅读 UI；两者可随时切换 |
| F10 | **后台播放** | 应用切换到后台或锁屏后继续朗读；系统通知栏显示播放控制 |
| F11 | **预合成/离线模式** | 用户可选择某本书"全部预合成"；将 TTS 音频缓存到本地；预合成后离线收听 |
| F12 | **日轻专用分析** | 针对日本轻小说的增强角色识别：利用一人称代名詞（俺/僕/私/あたし）和語尾（～ぜ/～わ/～っす）特征；提升无主语对话的识别精度 |

#### P2 — 增强功能

| 编号 | 功能 | 描述 |
|:----:|------|------|
| F13 | **书架管理** | 书架界面：封面/书名/作者/上次阅读时间/进度；搜索和排序 |
| F14 | **音色市场/分享** | 用户可导出角色音色配置为 JSON；导入他人分享的角色配置；热门书籍的社区音色方案 |
| F15 | **情绪感知朗读** | LLM 标注对话情绪标签；TTS 根据情绪动态调整语调（开心/悲伤/愤怒/平淡） |
| F16 | **多语言支持** | 界面国际化 (中文/English/日本語)；跨语言小说支持 |
| F17 | **音频导出** | 将完整的有声书导出为 M4B/MP3 文件；含章节标记和元数据 |

### 3.3 功能详细说明

#### F1：EPUB/TXT 导入解析

```
输入: .epub 或 .txt 文件路径
处理:
  1. 检测文件格式
  2. EPUB: 用 epub_pro 解析 → 提取元数据(书名/作者/封面) → 按 spine 顺序遍历章节
     → 每个章节去除 HTML 标签 → 保留段落结构 → 输出纯文本
  3. TXT: 按编码检测 → 自动按章节标记(第X章/Chapter X/^第.{1,5}章$)分段
     → 无章节标记则按固定字数(5000字)分块
输出: Book 对象 {title, author, cover?, chapters: [Chapter{title, text}]}

大文件优化 (>50万字):
  - 首次解析仅加载章节元数据 (标题 + 起始偏移)，不加载全文
  - 进入章节时才加载该章文本
  - 惰性加载: 当前章节 + 前后各 1 章预加载，其余按需读取

错误处理: 文件损坏 → 提示用户；编码无法识别 → 尝试常见编码列表
```

#### F2：LLM 角色分析

```
输入: Book 对象
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

阶段1 (角色发现):
  取前3章(每章取前2000字) → 拼接 → 发送 LLM 请求
  Prompt: 角色发现 + 属性推断 + 音色描述生成 + 别名识别
  输出: {characters: {name: {canonical_name, aliases[], gender, age,
         personalities, voice_prompt}}}

阶段2 (逐章对话标注):
  对每个章节: 发送文本 + 已知角色列表
  Prompt: 逐行标注说话人 + 情绪 + 类型 + 指代消解
  输出: {script: [{speaker, line, emotion, type, voice_variant_id?}]}

阶段3 (增量角色发现，与阶段2并行):
  阶段2 每章标注中，遇到无法匹配已知角色的说话人 → 标记为 "unknown_N"
  该章完成后，若累计 unknown_N ≥ 2 或出现 3 次以上 → 触发增量分析
  LLM 分析 unknown_N 的上下文 → 生成新角色信息 → 合并入角色列表
  → 回补前文章节中该角色的 unknown_N 标记

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
核心机制:

1. 角色指代消解:
   - 同一角色的多个称呼 (本名、尊称、昵称、绰号) 合并为规范名
     例: "林风" = "林公子" = "小林" = "风儿"
   - 输出 canonical_name + aliases[] 字段
   - 无明确说话人的对话，根据上下文 (上一句说话人、对话内容、
     语气特征) 推断归属

2. 声线变体检测 (P0 基础 / P2 完整):
   检测角色在不同场景下的音色变化:
   ┌─────────────┬──────────────────────────┬──────────┐
   │ 类型        │ 示例                     │ 实现     │
   ├─────────────┼──────────────────────────┼──────────┤
   │ 语气变异    │ 阴阳怪气、嘲讽、撒娇      │ P0 情绪  │
   │ 伪装变声    │ 压低声音装男声、模仿他人   │ P2       │
   │ 生理状态    │ 受伤虚弱、醉酒、沙哑      │ P2       │
   │ 时间成长    │ 少年→成年、沧桑变化       │ P2       │
   └─────────────┴──────────────────────────┴──────────┘

   输出 voice_variant 字段:
   {
     variant_id, label, voice_prompt,      // 该变体的音色描述
     trigger_condition,                     // 触发条件 (P2)
     chapters: [int]                        // 出现的章节范围
   }

   ScriptLine 中引用:
   {speaker_id, voice_variant_id?, emotion} // variant_id=null 使用默认音色

   Prompt 指令要点:
   - 对话描述中提到角色改变说话方式 → 标记 voice_variant
   - 角色处于特殊生理状态 (受伤/醉酒/极度疲惫) → 标记 voice_variant
   - 跨章节时间跳跃 → 分段使用不同 age/voice_prompt
   - 无特殊标记时使用默认 voice_prompt

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
LLM 参数:
  model: 用户配置 (默认 qwen3.5-flash)
  temperature: 0.3 (低随机性，保持一致性)
  response_format: json_object
  max_tokens: 4096

日轻增强 (F12):
  当用户标记书籍为"日本轻小说"时，Prompt 中加入:
  - 一人称代名詞分析 (俺→男/自信, 僕→男/温和, 私→女/正式, あたし→女/活泼)
  - 語尾模式分析 (〜ぜ/〜ぞ→热血男, 〜わ/〜ですわ→优雅女, 〜っす→运动系)
  - 口調整体风格推断

错误处理:
  LLM API 不可达 → 提示用户检查网络/API 配置 → 允许重试
  返回 JSON 解析失败 → 重试一次 → 仍失败则降级为单音色模式
  Token 超出限制 → 自动缩小文本窗口
  增量角色发现产生的 unknown_N 超过 20 个 → 暂缓分析，提示用户
    "检测到大量未识别角色，可能文本格式不适合自动分析"
```

#### F3：角色音色配置存储

```
存储内容:
  1. analysis_cache/{book_hash}.json  — 角色配置 + 脚本
  2. tts_cache/{book_hash}/           — 预合成音频文件 (可选，F11)

book_hash = SHA256(文件路径 + 文件大小) 的前 16 位

缓存策略:
  打开书籍 → 计算 hash → 查找缓存
  命中 → 直接使用缓存 (秒进)
  未命中 → 触发 LLM 分析 → 分析完成自动存储

缓存失效:
  文件被修改 (大小或修改时间变化) → 提示用户重新分析
  用户手动选择"重新分析"
```

#### F4：多角色朗读

```
流程:
  1. 根据当前阅读位置，获取 ScriptLine
  2. 查表获取 speaker 对应的 CharacterInfo（含 voice_prompt）
  3. 调用 Qwen TTS API:
     POST /v1/audio/speech
     {model, input: line.text, voice: voice_prompt, speed, language}
  4. 接收音频 → 播放
  5. 播放同时预取下一句 (后台合成)
  6. 播放完毕 → 自动播放下一句 → UI 高亮移动到下一句

性能优化:
  - 预取窗口: 当前句播放时，后台合成下 2 句
  - 音色缓存: VoiceDesign 生成的 speaker embedding 在 TTS 服务端缓存
  - 短句合并: 同一角色连续短对话合并为单次 TTS 请求
    合并规则:
    · 同一 speaker_id 的连续 ScriptLine（中间无旁白或他人插话）
    · 累计 ≤ 100 字符 或 ≤ 3 句 → 一次 TTS 调用，句间插入自然停顿标记
    · 超过阈值 → 分组合成，每组保持语气连贯
    ScriptLine 增加 mergedGroupId 字段标记合并组
  - 句间过渡: 角色切换时插入自适应静音间隙
    · 默认过渡: 300ms
    · 情绪紧张/快速对话: 150ms (紧凑感)
    · 场景切换/章节过渡: 600ms (段落感)
    · 同一角色连续句合并组内: 使用 TTS 自然停顿 (SSML <break time="200ms"/>)

文本高亮:
  - 阅读视图中，当前正在朗读的句子以高亮背景显示
  - 同一角色的连续句子可使用角色关联的淡色标记
```

#### F5：播放控制

```
功能:
  - 播放/暂停: 空格键或点击播放按钮
  - 上一句/下一句: ← → 方向键或按钮 (回溯时重新合成，不缓存历史音频)
  - 快进/快退: 长按按钮 ±10 秒 (粗粒度) 或拖动进度条 (细粒度)
  - 倍速: 0.5x / 0.75x / 1.0x / 1.25x / 1.5x / 2.0x / 3.0x
    通过 TTS API speed 参数实现，非音频拉伸
  - 章节跳转: 章节下拉列表 → 跳转至目标章节首句
  - 进度指示: 当前句号/总句数 + 进度条 (含章节刻度标记)

键盘快捷键 (桌面端):
  Space        播放/暂停
  ← / →       上一句/下一句
  Ctrl+←/→    上一章/下一章
  ↑ / ↓       加速/减速 0.25x

错误处理:
  跳转目标句 TTS 合成失败 → 自动跳过至下一句 → 连续失败 3 句则暂停并提示
```

#### F6：阅读进度保存

```
自动保存触发时机:
  - 暂停播放时
  - 切换章节时
  - 应用进入后台/失去焦点时
  - 每播放 30 句自动保存一次 (防崩溃丢失)
  - 用户手动退出阅读器时

保存内容:
  {bookId, chapterIndex, lineIndex, positionFraction, updatedAt}

恢复策略:
  打开书籍 → 读取 ReadingProgress
  若 positionFraction > 0 → 弹出提示 "恢复到上次位置？[是] [从头开始]"
  默认 "是" 3 秒后自动确认

冲突处理:
  缓存被清空但进度存在 → 进度仍有效 (指向未缓存的 ScriptLine)
  书籍文件被修改导致章节结构变化 → positionFraction 作为 fallback 近似定位
```

### 4.1 性能

| 指标 | 10 万字 | 100 万字 | 500 万字 | 测量方法 |
|------|:------:|:------:|:------:|----------|
| EPUB 解析时间 | < 2 秒 | < 10 秒 | < 30 秒 | 计时 |
| LLM 角色分析 | < 60 秒 | < 10 分钟 | < 45 分钟 | 计时 |
| 缓存命中后打开书籍 | < 1 秒 | < 2 秒 | < 5 秒 | 计时 |

| TTS 单句合成延迟 | < 3 秒 | < 3 秒 | < 3 秒 | P50 延迟 |
| 预取命中率 | > 90% | > 90% | > 90% | 统计 |
| 应用冷启动时间 | < 3 秒 | — | — | 计时 |

**大文件优化策略**（>50 万字）：
- **分块加载**：章节按需加载，内存中仅保留当前章节 + 前后各 1 章，其余惰性加载
- **渐进式分析**：LLM 分析按章流式进行，已分析的章节立即可朗读，无需等待全书完成
- **脚本分片存储**：每章 ScriptLine 独立缓存文件，避免单文件过大（详见 §7.2）

### 5.3 关键技术决策

| 决策 | 选择 | 理由 |
|------|------|------|
| LLM 调用方式 | OpenAI 兼容 API | 可接入任何兼容 LLM，不锁定供应商 |
| TTS 调用方式 | OpenAI 兼容 API (`/v1/audio/speech`) | 可本地 Docker 也可云端，统一接口 |
| 角色分析时机 | 打开书籍后立即全量预处理 | 用户体验最简单，朗读时无等待 |
| 音频播放模式 | 逐句合成 + 预取 | 平衡延迟和灵活性，支持实时音色切换 |
| 跨平台策略 | Flutter-OH (社区版) | HarmonyOS 最成熟的跨平台方案 |
| 文本存储 | 本地纯文本 (EPUB→text 提取) | 隐私优先；LLM 分析结果缓存为 JSON |

**平台兼容性风险与缓解：**

| 风险点 | 详情 | 缓解 |
|--------|------|------|
| Linux 音频后端 | just_audio 在 Linux 上依赖 GStreamer，部分发行版默认未安装 | Fallback 到 audioplayers；文档标注 `apt install gstreamer` 依赖 |
| OH 音频支持 | Flutter-OH 的音频插件生态不如官方 Flutter 成熟 | 备选 audioplayers_ohos；或自研 OH 音频 Platform Channel |
| Flutter-OH 版本滞后 | 社区版通常滞后官方 Flutter 1-2 个月 | 锁定 Flutter 3.27，避免使用最新 API |
| Isar OH 兼容 | Isar 对 OH 平台的支持状态需验证 | 备选 drift (SQLite) 或 Hive |

### 6.3 核心交互流程

```
首次使用:
  [启动应用] → [设置页: 配置 LLM/TTS API] → [书架(空)] → [导入EPUB]
  → [分析进度条: "正在分析角色..."] → [分析完成] → [自动进入阅读器]
  → [自动开始朗读]

后续使用:
  [启动应用] → [书架: 显示历史书籍] → [点击书籍] → [缓存命中]
  → [秒进阅读器，定位到上次位置] → [点击播放]

  [阅读器] → [🔊角色按钮] → [角色列表] → [点击角色]
  → [编辑音色描述] → [试听] → [满意] → [返回继续听]

预合成:
  [书架] → [长按书籍] → [预合成] → [后台合成进度] → [完成通知]
```

### 6.4 设计原则

1. **沉浸式阅读优先** — 控制条和 UI 元素尽量不遮挡文本，阅读器为绝对核心
2. **即开即听** — 默认行为是导入书后自动开始分析+朗读，减少用户操作步骤
3. **角色可见** — 当前正在说话的角色始终在界面可见位置（底部指示条），增强"多角色"感知
4. **渐进复杂度** — 默认全自动（用户无需了解 LLM/TTS），高级选项折叠在设置中
5. **隐私可见** — 设置页面明确展示数据流向（哪些内容发送到哪个 API）

## 7. 数据模型

### 7.1 核心实体

```
Book (书籍)
  id: String (hash)
  title: String
  author: String?
  coverPath: String?
  filePath: String
  fileType: enum { epub, txt }
  chapterCount: int
  addedAt: DateTime
  lastOpenedAt: DateTime?

Chapter (章节)
  index: int
  title: String
  text: String

CharacterInfo (角色)
  id: String (uuid)
  bookId: String (外键)
  canonicalName: String            # 规范名 (如 "林风")
  aliases: List<String>            # 别名 (如 ["林公子", "小林", "风儿"])
  gender: enum { male, female, unknown }
  age: enum { child, teen, young_adult, middle_aged, elderly, unknown }
  personalities: List<String>     # 性格标签
  firstPerson: String?            # 一人称 (日轻)
  sentenceEndings: List<String>?  # 語尾特征 (日轻)
  voicePrompt: String             # VoiceDesign 中文音色描述 (默认)
  voiceVariants: List<VoiceVariant>?  # 声线变体 (P2 完整实现)
  isNarrator: bool                # 是否为旁白

VoiceVariant (声线变体)
  variantId: String
  label: String                   # 如 "受伤虚弱", "伪装男声", "少年时期"
  voicePrompt: String             # 该变体的音色描述
  triggerCondition: String?       # LLM 推断的触发条件描述
  chapters: List<int>?            # 出现的章节范围 (时间成长型)

ScriptLine (脚本行)
  id: int
  bookId: String
  chapterIndex: int
  lineIndex: int                  # 在章节中的序号
  speakerId: String               # 角色 ID (旁白为 "narrator")
  voiceVariantId: String?         # 声线变体 ID (null = 默认音色)
  text: String                    # 原文本
  emotion: String?                # 情绪标签
  type: enum { narration, dialogue, monologue }
  mergedGroupId: int?             # 短句合并组 ID (同一组连续合成)

ReadingProgress (阅读进度)
  bookId: String (主键)
  chapterIndex: int
  lineIndex: int
  positionFraction: double        # 0.0-1.0 全书进度
  updatedAt: DateTime

ApiConfig (API 配置)
  id: int (主键, 单例)
  llmBaseUrl: String
  llmApiKey: String
  llmModel: String
  ttsBaseUrl: String
  ttsApiKey: String
  ttsModel: String
```

### 7.2 缓存文件结构

```
{cache_dir}/
├── analysis/
│   └── {book_hash}/
│       ├── meta.json              # 角色列表 + 书籍元数据
│       ├── ch_0001.json           # 第 1 章脚本 (ScriptLine[] + 新发现角色)
│       ├── ch_0002.json           # 第 2 章脚本
│       └── ...                    # 每章独立文件，支持渐进式分析和惰性加载
├── audio/
│   └── {book_hash}/
│       ├── {chapter}_{line}.mp3   # 预合成音频 (F11)
│       └── ...
└── config/
    └── api_config.json            # API 配置 (加密存储)
```

### 7.3 Schema 迁移策略

所有持久化 JSON 文件均包含 `version` 字段，确保跨版本兼容。

```
迁移规则:
  读取文件 → 检查 version:
    = 当前版本 → 正常加载
    < 当前版本 → 运行对应迁移函数链 (如 v1→v2→v3)
    > 当前版本 → 提示 "数据由更新版本创建，请升级应用"

迁移函数注册:
  migrations = {
    1→2: migrateV1ToV2,   // 示例: name → canonicalName + aliases
    2→3: migrateV2ToV3,   // 示例: 增加 voiceVariants 字段
  }

迁移失败处理:
  保留原文件备份 (*.bak) → 提示用户 → 触发重新分析 (LLM 缓存可重建)
  进度数据尝试抢救: 若仅角色/脚本迁移失败，进度数据仍可独立加载
```

#### 9.4.6 关键结论

- **LLM 分析成本极低**：最便宜的 Qwen3.5-Flash 仅 ¥0.12/本（10 万字），最贵的 MiniMax M2.5 也仅 ¥0.65/本。且为**一次性开销**，缓存后永久复用。

- **本地部署零 API 费用**：使用 Ollama + 本地 Docker TTS 仅需电费和算力。
- 原 "~¥0.05/本" 仅覆盖了 LLM 分析成本且偏乐观，实际单书 LLM 分析约 **¥0.12–0.65**，TTS 收听另计。
